# Plan: Fix four gameplay bugs found in manual playtesting

## Goal

Fix four bugs surfaced this session via hands-on `/run` testing and user
follow-up: other players' text inputs get wiped when one player submits,
the drawing canvas lags one stroke behind on color/width changes, the
host doesn't see book replays on Reveal, and the Moderation Panel's kick
control doesn't kick anyone.

## Scope

In scope: the four bugs below, each as a `[ui]`-tagged fix with a
regression test. Out of scope: anything from `.project/audit.md` (a
separate, still-open set of design questions/risks — not actioned by this
plan) and the constitution's still-declined performance-budget defect
(unrelated, unaffected by this work).

## Technical Approach

### F1 — phrase/drawing input cleared for other players (`feedback-main-3ea6.md`)

Root cause, confirmed by reading `WritingDrawing.svelte:51-54`:

```svelte
$: if (myTurn) {
  textValue = '';
  drawnOps = [];
}
```

`myTurn` is `computeNextEntries(state.room).find(...)` — a **new object
reference every time `state.room` updates**, which happens on *every*
room broadcast (e.g. whenever any other player submits, not just when
this player's own assigned turn changes). Svelte's `$:` re-runs whenever
any reactive dependency's value is recomputed, and since `myTurn` is a
fresh object each broadcast, this block fires and wipes local draft
state on every unrelated room update, not only on an actual turn
reassignment for this player.

Fix: key the reset off something identity-stable — the previous vs.
current turn's `bookId` + `position` (or simplest: only reset when
`myTurn` transitions from undefined to defined, i.e. a *newly assigned*
turn, not on every recomputation of an already-assigned turn). Track the
previous turn's book/position in a local variable and only clear
`textValue`/`drawnOps` when that identity actually changes.

### F2 — drawing canvas shows stale color/width mid-stroke (`feedback-main-6d3d.md`)

Root cause, confirmed by reading `DrawingCanvas.svelte`: `ctx.strokeStyle`
and `ctx.lineWidth` are set once in `onMount` (line 154-158) and
thereafter only inside `redrawAll()` (lines 92-93), which runs per
*completed* op after `onOpsChange` fires. `drawSegment()` (called live
from `handlePointerMove`, line 115-121) never sets `ctx.strokeStyle`/
`ctx.lineWidth` itself — it draws with whatever the canvas context's
style was last set to, which is stale until the in-progress stroke
finishes and `redrawAll()` re-applies each op's *own* color/width.
`selectColor`/`selectWidth` (lines 134-140) only update component state,
never touch `ctx` directly.

Fix: set `ctx.strokeStyle = effectiveColor` and `ctx.lineWidth =
activeWidth` in `handlePointerDown` before starting a new stroke (and/or
reactively whenever `effectiveColor`/`activeWidth` change while no
stroke is in progress), so a newly-started stroke always draws with the
currently-selected values from its first segment.

### F3 — host doesn't see book replays on Reveal (`feedback-main-4258.md`)

Inspected `Reveal.svelte`, `App.svelte`, and the relevant server handlers
(`onSubmitEntry`, ack-vs-`socket.to()` broadcast pattern) — nothing in
this code path is conditioned on `isHost`, and the ack/broadcast split
(sender gets the updated room via ack, everyone else via broadcast) is
symmetric and matches the documented pattern in `infrastructure.md`. No
root cause could be confirmed by static reading alone; this needs
reproduction. Task T005 below is a reproduce-first task: drive the app
with `/run`-style multi-session testing (host and 2+ non-host players)
through to `reveal`, and instrument/observe specifically what differs for
the host's client (room state received? component mount timing? `isHost`
derivation?) before writing a fix.

### F4 — Moderation Panel "kick" does nothing (`feedback-main-e2ff.md`)

Inspected `ModerationPanel.svelte`, `session.ts`'s `kickPlayer`, and
`onKickPlayer` in `handlers.ts` — the wiring (button → `session.kickPlayer`
→ `emitWithAck('kickPlayer', ...)` → `onKickPlayer` → `target.kicked =
true` → ack + broadcast) reads correctly, and `DEFECTS.md`'s last full
pass found this handler matching its artifact description with passing
tests. No root cause could be confirmed by static reading; this needs
reproduction, same as F3. Task T006 is reproduce-first: drive a real kick
through `/run` and observe where the flow actually breaks (is
`isHost` true? does the ack arrive? does `room.players` update? is the
button even wired to the visible instance of `ModerationPanel`, given it's
mounted separately from three different parent views — Lobby,
WritingDrawing, Reveal?).

## Open Questions

- F3 and F4's actual root causes are unknown until Phase 2's reproduction
  work happens — the Technical Approach's static-reading pass ruled out
  the obvious candidates (isHost gating, ack/broadcast asymmetry) for
  both, so the real cause may be more subtle (e.g. a stale closure, a
  session/reconnect edge case, or something specific to how multiple
  local sessions were tested this session with manually-cleared
  `localStorage` rather than real separate browser profiles — worth
  ruling out as a testing artifact vs. a real production bug during
  T005/T006).
