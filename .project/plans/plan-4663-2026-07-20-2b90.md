# Plan — onStartGame counts active players

## Goal

Make `onStartGame` resolve both the minimum-player check and the
laps-per-book default from the active (non-kicked) roster, so a
lobby-kick can neither slip a below-minimum game past the gate nor shorten
the game's lap count.

## Scope

**In scope.** The two `room.players.length` reads in `onStartGame`
(`server/src/socket/handlers.ts`): the minimum-player gate (`:154`) and
the `defaultLapsPerBook` resolution (`:160`). Both switch to the active
count via the existing `activePlayers` helper. Plus the two
`datamodel.md` normalization rules that describe these counts without
settling whether a kicked lobby player is "live" — Minimum player count,
and the Laps-per-book live default.

**In scope by the user's explicit decision this run:** the minimum-player
gate. The filed feedback (F001) named only the lap default; reading the
code showed the line above it has the identical bug, and the user chose
to fix both as one coherent change.

**Not in scope.** Any other `room.players.length` read elsewhere in the
codebase — the mid-game rotation and completion sites were already
routed through `activePlayers` by the `plan-25a0` defect pass, and this
plan does not revisit them. No change to whether/when a kick is permitted
during `lobby`.

## Technical Approach

`activePlayers(room)` already exists in `shared/src/turnAdvancement.ts`
(introduced by `plan-25a0`) and is the single source of "who counts."
Both `onStartGame` sites currently pre-date it. The change is to compute
`const active = activePlayers(room).length` once at the top of the
start-game path and use it for:

- the minimum-player comparison (`active < MINIMUM_RECOMMENDED_PLAYERS`),
  so the `acknowledgeSmallGame` override is required exactly when the
  *real* remaining roster is below the floor; and
- `defaultLapsPerBook(active)`, so the live default matches the game that
  actually plays.

This is the same seam `plan-25a0` closed for rotation and vote
membership, extended to the one place a game is *started* rather than
advanced. No new abstraction — it reuses the helper the earlier pass
already justified.

One behavioral note worth stating because it is a real (correct) change,
not just a refactor: after this, kicking a lobby down to 2 real players
and starting **will** demand the small-game acknowledgement, where today
it silently starts a 3-counted game. That is the intended fix, not a
regression — but it is a user-visible change to when the override prompt
appears, so its test should pin it explicitly.

## Phase Breakdown

Single phase — the change is small and the two sites are one edit.

### Phase 1 — Active-count resolution in onStartGame

- Resolve `activePlayers(room).length` once in `onStartGame` and use it
  for both the minimum-player gate and `defaultLapsPerBook`. `[F001]`
  `[artifacts: datamodel]`
- Tests: (a) a 5-player lobby kicked to 4 then started resolves to the
  2-lap default, not 1; (b) a 3-player lobby kicked to 2 then started
  without `acknowledgeSmallGame` is rejected `too-few-players`, and
  succeeds with the acknowledgement; (c) an all-present lobby behaves
  exactly as before (regression guard).
- Correct `datamodel.md`'s **Minimum player count** and **Laps per book**
  normalization rules to state the count is of active (non-kicked)
  players, resolving the "live player count" ambiguity the feedback
  flagged. `[F001]` `[artifacts: datamodel]`
