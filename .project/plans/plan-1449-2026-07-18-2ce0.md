# Plan: Finish the salon/gallery reskin beyond GiltFrame

## Goal

Apply the salon/gallery theme (color tokens, Fraunces/Rubik type) across
every view's page chrome — backgrounds, headings, body text, labels,
borders, and controls — closing the gap where only `GiltFrame` itself
got reskinned and everything else kept its pre-redesign default
Tailwind slate-gray/white styling.

## Scope

**In scope:**
- A global base layer (page background, base text color, base body
  font) so every screen inherits the theme by default rather than
  relying on every element remembering a class.
- Per-file replacement of every remaining default-Tailwind slate/white
  utility class with the corresponding theme token, across
  `Lobby.svelte`, `WritingDrawing.svelte`, `Reveal.svelte`,
  `ModerationPanel.svelte`, `TurnStatus.svelte`, and `App.svelte`'s
  terminal-state `<main>` blocks.
- Page headings ("Exquisite Telephone," "Reveal," section headers) set
  to the display face (Fraunces).
- Primary/secondary button styling made consistent with the palette
  (bubblegum primary, marigold/butter secondary) everywhere a plain
  bordered gray button currently exists.
- A `ui.md` clarification recording that the theme applies globally, not
  only to `GiltFrame`, plus the slate-to-token mapping convention below
  so this gap can't quietly recur.
- A regression-guard test per touched file: a source-scan assertion
  that the file contains no `slate-*`/plain `bg-white` utility classes
  after the fix, so "partially reskinned" fails CI rather than shipping
  quietly again.

**Out of scope:**
- Functional status colors — error alerts (red), timer/vote-prompt
  warnings (amber), success confirmations (green/grass) — these encode
  meaning, not decoration, and are already reasonably themed
  (`grass` already doubles as the success color); only truly
  decorative/neutral chrome (slate grays, plain borders, default
  buttons) is in scope.
- `DrawingCanvas.svelte`'s actual drawing surface (`bg-white`) — a
  plain white "page" to draw on is the correct, expected surface, not
  leftover chrome; its toolbar was already reskinned by the prior plan.
  Only its toolbar is touched here if any slate remnants are found.
- Any new component, layout restructuring, or copy changes — this is a
  styling-only pass over what `salon-gallery-ui-redesign` already built.

## Technical Approach

**Global base layer** (`client/src/app.css`): add a Tailwind `@layer
base` block setting `body` to `bg-butter text-ink font-body`, and a rule
applying `font-display` to `h1, h2`. This gives every screen a correct
default background/font/color even before any per-element class is
touched — but per-element Tailwind utility classes (e.g. an explicit
`text-slate-700`) have higher specificity than the `body` element
selector and will still win, so this alone doesn't fix already-styled
text; it's the safety net for anything unstyled, and the base case for
new markup going forward.

**Per-file slate-to-token mapping** (applied explicitly, file by file,
since the base layer can't override existing utility classes):

| Current | Replace with |
|---|---|
| `text-slate-900` / `text-slate-800` | `text-ink` |
| `text-slate-700` | `text-ink/90` |
| `text-slate-600` | `text-ink/75` |
| `text-slate-500` | `text-ink/60` |
| `text-slate-400` | `text-ink/45` |
| `border-slate-200` (decorative dividers/card borders) | `border-marigold/30` |
| `bg-slate-800` (primary buttons — e.g. Lobby's "start game") | `bg-bubblegum` (matching the pattern already used in `WritingDrawing`/`Reveal`) |
| `bg-slate-100` (Reveal's cover-art placeholder box) | `bg-butter` |
| plain unstyled/gray secondary buttons (`border` with no color) | `border-marigold/60 bg-butter text-ink` |

Headings (`<h1>`/section `<h2>`-equivalent) additionally get
`font-display`. This table is also added to `ui.md`'s Visual Identity
section so it's a durable, referenceable design decision, not something
that only lives in this plan.

**Regression guard**: each touched `*.test.ts` file gets one additional
test — read the corresponding `.svelte` file's source text
(`fs.readFileSync`) and assert it contains no `slate-` substring (and,
for `DrawingCanvas.svelte`, no `bg-white` outside the canvas element
itself — scoped by only scanning the toolbar's template region, or by
asserting the specific toolbar button classes don't use it, implementer's
call for the least-brittle assertion shape). This is a plain source-text
assertion, not a rendered-DOM assertion, and is what makes "silently
reskin only half the file" fail loudly instead of passing review by
looking plausible.

## Open Questions

- Exact opacity values in the mapping table (`/90`, `/75`, `/60`,
  `/45`) are a reasonable starting point but not pixel-tested — if a
  contrast issue turns up during implementation (e.g. `text-ink/45` too
  light against `bg-butter`), adjusting the specific opacity is an
  implementer's call within the spirit of the mapping, not a plan
  deviation.
- Whether every one of the four App.svelte terminal states (T007)
  actually needs its own `font-display` heading, or just uses
  `font-body` at slightly larger size — none of them currently have a
  distinct `<h1>`, just a single `<p>` — is left to T007's
  implementation; the mapping table's text-color rule applies regardless.
