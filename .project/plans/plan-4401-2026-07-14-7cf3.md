# Plan: Fix observability gap in onEndGame

## Goal

Make `onEndGame`'s host-triggered game completion emit a structured
`game_completed` log event, matching the natural-completion path's
behavior and closing the Principle IX gap recorded as defect
`4056be9c`.

## Scope

**In:**
- `onEndGame` (`server/src/socket/handlers.ts`) logs a `game_completed`
  event on the host-ended path, distinguishable from natural completion
  via a `reason: 'host-ended'` field (natural completion, in
  `onSubmitEntry`, has no `reason` field — its absence already implies
  "completed by finishing the last entry")

**Out (declined/handled without a task, this run):**
- Defect `17a7ea0a` (missing performance budgets for real-time
  operations) — declined by the user this run; recorded in
  `surfaced-defects` so it won't be re-prompted, but no fix task here.
- Defect `a5b7918a` (infrastructure.md's Realtime Sync section naming
  a nonexistent `onDrawStroke` handler) — a documentation-only
  correction, already applied directly to `infrastructure.md` as part
  of drafting this plan (no code involved, so no `/ardd-implement`
  task needed for it).

## Technical Approach

`onEndGame` currently doesn't accept a `Logger` parameter at all. Add
one (mirroring every other handler's signature — `onStartGame` is the
closest analog: host-only guard, no other side effects besides the
status transition and broadcast). Log
`{ event: 'game_completed', outcome: 'success', roomId, reason:
'host-ended' }` right after the `room.status = 'ended'` assignment,
before the broadcast. Update the `set('endGame', ...)` registration in
`server/src/socket/server.ts` to pass `logger` through, matching the
pattern already used for `onSubmitEntry`/`onCreateRoom`/etc.
