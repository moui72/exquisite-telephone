# Plan — clear the 2026-07-20 defect snapshot

## Goal

Close all seven recorded code-vs-artifact defects, making `Player.kicked`
consistently honored across turn rotation, vote membership, and roster
rendering, and making timer extensions actually extend.

## Scope

**In scope** — the seven `DEFECTS.md` entries surfaced this run:

| id | What |
|---|---|
| `30285161` | Restart doesn't restore continuability (turn rotation ignores `kicked`) |
| `bf503c53` | Timeout-vote membership ignores `kicked` |
| `4de54acc` | Kicked players still render in `Lobby`/`TurnStatus` |
| `e3de2e2a` | Timer extensions replace rather than add to base duration |
| `1e4c9016` | "Waiting for the round to finish" misfires past lap 1 |
| `2d6dda28` | Rejoin-a-dead-room screen has no return control |
| `d991281a` | Curation cap has no Production Annotation |

**Not in scope.** Draining or truncating the curation log — that is the
backlogged `curation-data-aggregation-pipe`'s job, and this plan only
documents the cap as an accepted shortcut pointing at it (user decision
this run). Also out: any change to the kick→`nonContinuable`→restart
*flow* itself; this plan makes restart deliver what it already promises,
not redesign moderation.

## Technical Approach

**The kicked-player seam is one change, not three.** `Player.kicked`
today is honored wherever a kicked player is a *source* of work
(`createBooksForRoom` book origins, `dealPrompts`) and ignored wherever
they are a *recipient* of it. The fix is a single shared notion of "the
room's active players" that rotation, vote membership, and roster
rendering all read from — introduced once in `shared/`, per Principle VII
(named types over inline duplication) and Principle VI (derived, not
persisted). Rotation is the load-bearing piece and lands first; the
other two consume the same helper.

Two subtleties the implementation must handle, both discovered by reading
the code rather than inferred from the defect text:

1. **A book whose origin author was kicked.** Kicking during `writing`
   sets `nonContinuable`, and the host is expected to restart (which
   regenerates books, excluding kicked origins) or end. But *existing*
   books in a not-yet-restarted room can still have a kicked origin. Once
   rotation indexes into an active-players list, `findIndex` on such a
   book returns `-1` and the arithmetic silently produces a wrong author.
   `computeNextEntry` must return `null` for those books explicitly
   rather than let the index underflow.
2. **Completion and lap defaults are player-count-derived.** Both
   `position >= players.length * laps` and `defaultLapsPerBook(count)`
   must switch to the active count, or a restarted room's books will
   never complete (they would wait on turns that are never assigned).

**Timer extensions become additive** (user decision this run): the
deadline becomes `roundStartedAt + base + (extensions ?? 0)` instead of
`roundStartedAt + (extensions ?? base)`. The accumulation site
(`timerExtensions[id] = (existing ?? 0) + extensionMs`) is already
correct and does not change. Because `datamodel.md` documents the
*current* replacing formula and `ui.md` documents the additive one, this
change makes the code match `ui.md` and requires `datamodel.md` to be
corrected — the reverse of the usual direction, and worth stating so the
artifact edit doesn't look like drift being introduced.

The remaining three are self-contained and independently testable.

## Phase Breakdown

### Phase 1 — Active-player rotation (no dependency)

The load-bearing fix; everything in Phase 2 reads what this introduces.

- Add an `activePlayers(room)` helper in `shared/` returning
  `room.players.filter((p) => !p.kicked)`, with tests.
- Rewrite `computeNextEntry` to rotate over active players: origin index
  found within the active list, `authorIndex` modulo active length,
  completion checked against `activeCount * laps`, and an explicit `null`
  for books whose origin author is kicked. `[defect: 30285161]`
- Point `defaultLapsPerBook`'s call sites at the active count.
- Test: a restarted room with a kicked player assigns turns only to
  remaining players and completes; a book with a kicked origin yields
  `null`.

### Phase 2 — Vote membership and roster rendering (depends on Phase 1)

- Exclude kicked players from `stalledPlayerIds`/`eligibleVoterIds` in
  `sweepRoom`, via Phase 1's helper. `[defect: bf503c53]`
- Filter kicked players out of `Lobby` and `TurnStatus` rendering, so the
  artifact's "host **and other players**" claim holds. `[defect: 4de54acc]`
- Test both; the roster test should assert a kicked player is absent from
  a non-host view, not merely that the Moderation Panel still filters.

### Phase 3 — Timer extensions become additive (no dependency)

- Change `deadlineFor` (server) and the `deadline` derivation (client) to
  add the extension to the base duration rather than replace it.
  `[defect: e3de2e2a]`
- Correct `datamodel.md`'s `Room.timerExtensions` description and its
  stated deadline formula to the additive rule.
  `[artifacts: datamodel]`
- Test: granting an extension moves the deadline *later*, and a 30m timer
  plus 15m yields 45m — the regression the current code inverts.

### Phase 4 — Remaining UI drift (no dependency)

- Fix the "waiting for the round to finish" condition to compare against
  `activeCount * laps` rather than one lap, so it holds for the whole
  game. `[defect: 1e4c9016]`
- Add the documented return control to the rejoin-a-dead-room branch in
  `App.svelte`. `[defect: 2d6dda28]`
- Test both.

### Phase 5 — Curation cap annotation (no dependency)

- Add the 65,536-event cap to `infrastructure.md`'s Production
  Annotations, stating that events are dropped rather than rotated, that
  a curator therefore holds the *first* N ratings, and that
  `curation-data-aggregation-pipe` is the intended remedy.
  `[defect: d991281a]` `[artifacts: infrastructure]`
- No code change.

## Complexity Tracking

Required by Principle I (complexity must be justified).

| Deviation | Why justified | Simpler alternative rejected because |
|---|---|---|
| A shared `activePlayers` helper rather than inlining `.filter(p => !p.kicked)` at each of the three sites | Three call sites across `shared`/`server`/`client` with identical semantics is the "genuine duplication" Principle I names as the trigger for an abstraction | Inlining is what produced this defect class in the first place — the filter already exists inline at `createBooksForRoom` and `dealPrompts`, and the sites that forgot it are exactly the bug |

No other deviation: every other change is a local edit to existing logic.

## Open Questions

1. **Should a kicked player's own in-flight book be dropped, or left
   frozen?** Phase 1 returns `null` for books with a kicked origin, which
   leaves them permanently incomplete in a not-yet-restarted room. That is
   correct for the frozen-room state, but if a future change lets play
   continue after a kick without a restart, those books would strand.
   Out of scope here; flagged because Phase 1 is where the assumption
   gets encoded.
2. **Does making rotation active-aware weaken the reason
   `nonContinuable` exists?** If rotation now skips kicked players
   correctly, a mid-game kick is arguably survivable without a restart.
   This plan deliberately does *not* relax the freeze — but the question
   is worth answering before someone concludes the freeze is now
   redundant.

- **Curation event cap (Phase 5)** — `MAX_CURATION_EVENTS = 65_536`;
  events past the cap are dropped, not rotated, so a curator holds the
  first N ratings rather than the most recent. Accepted for an app whose
  curation data is read by one human every few weeks; the remedy is the
  backlogged `curation-data-aggregation-pipe`. To be recorded under
  `infrastructure.md`'s `## Production Annotations` heading, per the
  constitution's single-location rule.
- No new shortcut is introduced by Phases 1–4.
