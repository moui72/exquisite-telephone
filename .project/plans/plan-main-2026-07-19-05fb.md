# Plan: fix drawing submit button and add white palette swatch

## Goal

Fix the unclickable drawing-submit button and give players a way to
erase/undo a fill by adding white to the drawing color palette.

## Scope

In scope:
- Reproduce and fix the drawing submit button's unclickable state in
  `WritingDrawing.svelte` / `DrawingCanvas.svelte`.
- Add white to `DrawingCanvas.svelte`'s `PALETTE_COLORS` so selecting it
  and stroking/filling paints back to the canvas's white background,
  per the `ui.md` update already applied.

Out of scope:
- A dedicated eraser tool or undo/redo stack (white-as-paint is the
  agreed mechanism, per `ui.md`'s Writing / Drawing View).
- Any other toolbar or palette redesign.

## Technical Approach

**F001 (submit button unclickable).** No static read of
`WritingDrawing.svelte`, `DrawingCanvas.svelte`, or `GiltFrame.svelte`
turned up an obvious cause (no `disabled` attribute, no overlay/
`pointer-events` rule, no z-index stacking issue) — the button markup,
click handler (`handleSubmitDrawing`), and CSS all look correct in
isolation. This means the fix task must start with live reproduction
(via `/run` or manual browser testing) to find the actual failure mode
before touching code — could be a runtime state issue (e.g. `myTurn`
becoming stale, a re-render race), a CSS issue only visible with real
layout/viewport sizing, or something else entirely. The task is written
test-first once the repro is understood, per Principle III below.

**F002 (white palette swatch).** Straightforward: add `'#ffffff'` to
`DrawingCanvas.svelte`'s `PALETTE_COLORS` array. The existing swatch
button markup (a plain color circle, `aria-label="Color {color}"`)
already handles arbitrary hex values with no special-casing needed —
white paints over existing strokes/fills the same way any other color
does, since the canvas background is white (`bg-white` in the `<canvas>`
class). One visual nuance: the white swatch button itself needs a
visible border against the toolbar's `bg-butter/60` background so it
doesn't disappear — the existing `border-2` ring already present on
every swatch (`border-velvet` when active, `border-transparent`
otherwise) needs a subtle non-transparent default border for the white
swatch specifically, so it's visible before selection too.

## Phase Breakdown

### Phase 1: Reproduce and fix the submit button (F001)
- T001 [artifacts: ui] Reproduce the drawing-submit-button click failure
  live (via `/run`, in a real browser) — get to the drawing step, attempt
  to click "Present your contribution" under the canvas, and record the
  actual failure mode (no click event firing, click firing but handler
  no-op, an exception, etc.). This determines what the fix and its test
  target.
- T002 [artifacts: ui] Based on T001's finding, write a failing
  regression test in `WritingDrawing.test.ts` or `DrawingCanvas.test.ts`
  that reproduces the bug, then fix it in
  `client/src/lib/views/WritingDrawing.svelte` and/or
  `client/src/lib/components/DrawingCanvas.svelte`. Depends on T001.

### Phase 2: Add white as a palette color (F002)
- T003 [artifacts: ui] [parallel] Add `'#ffffff'` to
  `PALETTE_COLORS` in `client/src/lib/components/DrawingCanvas.svelte`,
  plus a default visible border (e.g. `border-marigold/30`) on the white
  swatch specifically so it's visible against the toolbar background
  before selection, not just when active. Write/update a test in
  `DrawingCanvas.test.ts` asserting the white swatch renders and,
  when selected, new strokes/fills use `#ffffff`.

### Phase 3: Full-suite verification
- T004 Run the full test suite, typecheck, and lint across the monorepo;
  confirm all green. Confirm no regressions in existing
  `DrawingCanvas.test.ts` / `WritingDrawing.test.ts` coverage.
