# Plan — laps-aware book completion via a single source

## Goal

Make `onSubmitEntry`'s `book-complete` guard use the same laps-aware,
active-count completion condition as `computeNextEntry`, so multi-lap
games (including the default <5-player 2-lap game) can advance past lap 1
through the submission path.

## Scope

**In scope.** The `book-complete` guard in `onSubmitEntry`
(`server/src/socket/handlers.ts`) and the completion condition it
duplicates from `computeNextEntry` (`shared/src/turnAdvancement.ts`).
Defect `d27f4eea`.

**Not in scope.** The kicked-origin `originIndex === -1 → null` branch in
`computeNextEntry` (a separate, correct case — a book whose origin author
was kicked pre-restart returns `round-not-open`, unchanged here). The
Reveal play-again readiness count noted in `DEFECTS.md` under "not
defects" — that is a possible future `/ardd-feedback`, not part of this
fix.

## Technical Approach

The defect is fundamentally a **duplicated completion condition that
drifted**. `computeNextEntry` completes correctly at
`activeCount === 0 || position >= activeCount * laps`. `onSubmitEntry`
carries a second, older copy of "is this book done?" —
`book.entries.length >= room.players.length` — that predates both the
laps feature and the active-count work, so it caps every book at one raw
rotation and shadows the correct check that runs one line later.

The guard is not simply removable: it exists to return a distinct
`book-complete` ack, where `computeNextEntry` returning `null` maps to
`round-not-open` (it conflates "complete" with "round not open yet" under
round-gating). Deleting the guard would mislabel a genuinely finished
book as `round-not-open`.

So the fix is to make the completion condition **single-source**: extract
`isBookComplete(room, book): boolean` into `shared/src/turnAdvancement.ts`
(the `activeCount === 0 || position >= activeCount * laps` test), have
`computeNextEntry` use it for its first null-branch, and have
`onSubmitEntry`'s `book-complete` guard call it in place of the raw
`book.entries.length >= room.players.length`. One definition of "done,"
consumed by both — which is exactly the duplication Principle I names as
the trigger for a shared helper, and removing it is what prevents this
class of drift from recurring.

Round-gating is untouched: after the (now laps-aware) `book-complete`
guard, `computeNextEntry`'s `position > currentRoundFor(room)` branch
still returns `null → round-not-open` for a book ahead of the room-wide
round.

## Phase Breakdown

Phase lists are plan work-items, not live checklists — progress is
tracked in the linked tasks file.

### Phase 1 — single-source book completion

- Add `isBookComplete(room, book)` to `shared/src/turnAdvancement.ts`
  (`activeCount === 0 || book.entries.length >= activeCount * laps`,
  resolving `laps` and `activeCount` exactly as `computeNextEntry` does),
  and have `computeNextEntry` use it for its completion null-branch.
  `[defect: d27f4eea]`
- Replace `onSubmitEntry`'s `book.entries.length >= room.players.length`
  guard with `isBookComplete(room, book)`, preserving the `book-complete`
  ack. `[defect: d27f4eea]`
- Tests: a 3-player game with `lapsPerBook` defaulted to 2 accepts
  submissions through the start of lap 2 and only returns `book-complete`
  after `players.length * 2` entries; a `lapsPerBook = 1` game still
  completes after one rotation (regression); `isBookComplete` unit tests
  cover the boundary (`position` one below, at, and one above
  `activeCount * laps`) and the all-kicked `activeCount === 0` case.

## Complexity Tracking

Required by Principle I (complexity must be justified).

| Deviation | Why justified | Simpler alternative rejected because |
|---|---|---|
| Extract `isBookComplete` rather than inline the fixed condition at both sites | Two call sites already encode the same completion rule and one drifted — this is the "genuine duplication makes it unambiguous" trigger Principle I names for introducing an abstraction | Just correcting the guard in place fixes the symptom but leaves two independent copies of the completion condition free to drift again — the exact failure that produced this defect |

## Open Questions

None. The correct condition already exists in `computeNextEntry`; this
plan promotes it to a shared helper and routes the second call site
through it.
