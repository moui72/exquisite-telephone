# Plan: Fix flood-fill leaking across entries in PNG export

## Goal

Fix a bug where an unbounded background flood-fill on a drawing turn
leaks into other entries' rows in the exported PNG strip, instead of
staying confined to that turn's own entry.

## Scope

In scope: the one bug in `feedback-main-8a99.md`, as a `[infrastructure,
ui]`-tagged fix with a regression test. Out of scope: the live drawing
canvas's flood fill (`DrawingCanvas.svelte`/`floodFill.ts` itself) —
already correct and unaffected, since each turn draws on its own isolated
canvas; only the *export* composite is at issue. Also out of scope:
`ui.md`'s already-documented Production Annotation about exact-match-only
flood fill (color-distance tolerance at stroke edges) — that's a
different, already-accepted rough edge, not this bug.

## Technical Approach

Root cause, confirmed by reading `client/src/lib/export/pngExport.ts`
(`renderBookOntoContext`, lines 86-90) and `client/src/lib/drawing/
floodFill.ts`:

```ts
} else {
  const imageData = ctx.getImageData(0, 0, width, height);
  floodFill(imageData, { x: op.point.x, y: op.point.y + y }, op.color);
  ctx.putImageData(imageData, 0, 0);
}
```

`floodFill()` itself has no concept of entry boundaries — it fills every
contiguous same-color pixel within whatever `ImageData` it's handed
(`floodFill.ts:14-64`). During live drawing (`DrawingCanvas.svelte`),
this is safe because each turn renders onto its own isolated 320×240
canvas — there's nothing outside that entry's own drawing to leak into.
But PNG export (`pngExport.ts`) renders every entry in a book onto **one
shared composite canvas** stacked top-to-bottom (`renderBookOntoContext`
loops entries, incrementing `y` by each entry's row height). The `fill`
op branch passes `getImageData(0, 0, width, height)` — the *entire*
composite, not just this entry's own row (`[0, y, width,
DRAWING_ROW_HEIGHT]`) — so a background fill seeded on a still-white
background pixel with no enclosing stroke boundary within its own row
floods through every contiguous white pixel above and below, into
neighboring entries' rows (text rows and other still-unfilled drawing
rows are white background too).

Fix: scope the flood fill to just the current entry's own row. Extract
`ctx.getImageData(0, y, width, DRAWING_ROW_HEIGHT)` (row-local), seed the
fill with `op.point.x, op.point.y` (already row-relative — no `+ y`
offset needed since the sub-image starts at this row), then
`ctx.putImageData(imageData, 0, y)` to write it back at the correct
vertical offset. No change needed to `floodFill.ts` itself — it already
operates correctly against whatever bounds it's given; the bug is purely
in what `pngExport.ts` hands it.
